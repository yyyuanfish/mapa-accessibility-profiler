The extended architecture adds three layers not present in the current prototype. The Orchestration Layer grows from a single dispatcher into a structured layer with a MemoryRouter (session-state bus) and TraceLogger (audit trail for evaluation). The Profile Graph gains a MultimodalInputRouter, an UncertaintyQuantifier (Bayesian slot confidence), and a DialoguePolicyAgent (information-gain–driven question ordering, inspired by POMDP-based spoken dialogue management [Young et al., 2013]). The Planning Graph gains a SafetyAuditor (grounded claim verification), a CognitiveLoadAdapter (readability scoring and rewriting), and a MultimodalOutputRouter (modality dispatch: text, speech, sign gloss, map-only). Dynamic communication topology — which agent-to-agent links are active — is treated as a first-class design variable rather than a fixed hub-and-spoke structure [Jiang et al., 2026 Agent Q-Mix; Zhang et al., 2024 G-Designer; Yang et al., 2025 AgentNet].

The enriched profiling pipeline reframes dialogue as a constrained slot-filling problem over accessibility domains (mobility, vision, hearing, cognition), extending the frame-based dialogue tradition to a probabilistic setting [Young et al., 2013; Jurafsky and Martin, 2024].

NeedsExtractor (implemented): combines a deterministic lexicon scan over domain-specific keyword patterns with a structured LLM triage call. Both signals are merged by per-domain logical disjunction, and confidence is set to 0.85 for strong evidence and 0.30 for indirect evidence.

UncertaintyQuantifier (research contribution): rather than treating each slot as a binary yes/no, this node maintains a probability distribution over the slot's active value. If any route-critical slot falls below a confidence threshold, the dialogue is extended rather than closed prematurely.

DialoguePolicyAgent (research contribution): selects the next question by ranking open slots according to expected information gain — a simplified version of the POMDP belief-update and action-selection cycle. This is more principled than fixed-order questioning and reduces interaction length for users who have already revealed enough context.

MultimodalInputRouter (planned): routes text, speech, and image inputs to appropriate preprocessing branches before extraction. This is the integration point for camera-based hazard hints (consent-gated) and for voice-first interaction via faster-whisper [Systran, 2023; Radford et al., 2023]. The handling of visual input as supplementary rather than primary evidence is grounded in benchmarking evidence showing that even strong vision-language models achieve only ~16% task success on visually grounded web tasks, far below human performance, underscoring the importance of treating image analysis as advisory in safety-critical contexts [Koh et al., 2024 VisualWebArena].

The enriched planning pipeline adds three academically motivated components after the existing hazard_fusion_node.

SafetyAuditorAgent (research contribution): a post-fusion verifier that checks whether each accessibility claim in the candidate plan is grounded in the available evidence — route fixtures, ZüriACT barriers, or civic amenity data. Claims that are not supported by any retrieved evidence are removed or demoted to tentative warnings. This directly addresses the hallucination problem in safety-critical assistive navigation noted in the accessibility HCI literature [MacLeod et al., 2017]. The auditor can also loop back to hazard_fusion_node to request additional evidence before accepting a plan.

CognitiveLoadAdapter (research contribution): applies readability metrics (e.g., Flesch-Kincaid complexity) to the draft plan and, if the user profile indicates simple-language or memory-support needs, triggers a rewriting pass targeting shorter sentences and landmark-based cues. This operationalizes the Simple Language requirement as an automatic, measurable transformation rather than a rule-based string replacement.

MultimodalOutputRouter (research contribution): dispatches the final plan to one or more output channels based on the user's communication mode preference: plain text, speech-ready prose for TTS synthesis, sign-language gloss notation, or a map-only data bundle for screen-reader–friendly rendering. This makes the output-modality decision an explicit, auditable step rather than an implicit frontend rendering choice.

LiveTransitAgent (planned): integrates GTFS realtime feeds to annotate route steps with actual departure times, platform changes, and service disruptions at the point of plan generation.

The two-level structure of the planning pipeline — profiling intent at the higher level, route-step actions at the lower level — parallels the hierarchical LLM agent framework proposed by EPO, where separate models handle subgoal decomposition and low-level action generation, with preference signals derived from multimodal environment feedback used to align both levels [Zhao et al., 2024 EPO]. In MAPA, the profiling pipeline plays the role of subgoal generation (establishing what the user needs at the intent level), while the planning pipeline performs the corresponding action-level translation into concrete route steps and safety annotations. The SafetyAuditor and CognitiveLoadAdapter serve as post-hoc preference filters on the planning output, analogous to the reward-model-driven ranking step in EPO.

In the current prototype, the orchestrator uses a fixed hub-and-spoke topology regardless of user profile complexity. This is efficient for single-need profiles but wastes no communication on simple cases while missing cross-agent consultation that could benefit complex ones.

Inspired by work on communication topology as a design variable in multi-agent systems, the proposed extension introduces a topology selector that activates one of two pre-defined topologies at planning time based on the number of active accessibility domains. The most principled form of this idea is Agent Q-Mix, which reformulates topology selection as a cooperative MARL problem: each agent learns to choose from six communication actions (broadcast, query, debate, verify-with-tool, or work independently) using QMIX value factorization under a centralized-training / decentralized-execution (CTDE) paradigm, balancing task accuracy against token cost across the agent population [Jiang et al., 2026 Agent Q-Mix]. Earlier static topology design tools such as G-Designer and AgentNet provide complementary motivation for treating the communication graph as an explicit, tunable artifact [Zhang et al., 2024 G-Designer; Yang et al., 2025 AgentNet].

Topology A (single need, e.g., wheelchair only): a minimal sequential chain with no parallel branches and no uncertainty module. Optimised for latency.

Topology B (multi-need, e.g., wheelchair, low vision, simple language): the full graph is activated with parallel data branches, cross-agent consultation links (e.g., the NeedsExtractor notifies the AmenityLocator about mobility-relevant priorities), and the research-contribution nodes for safety auditing and cognitive adaptation.

The cross-consultation edges shown as dotted bidirectional links implement a restricted form of the agent message-passing described in AgentNet and Agent Q-Mix: agents can request a fact from a peer without routing the entire state back through the orchestrator, reducing redundant re-computation [Yang et al., 2025; Jiang et al., 2026].

Memory management is often treated as an implementation detail in multi-agent systems, but in accessibility navigation it has direct consequences for correctness and user autonomy.

Working Memory (session-scoped, volatile): holds per-turn slot values, confidence scores, dialogue history, and the agent trace log. It is the only memory tier active in the current prototype. It is cleared at session end and never leaves the server.

Persistent Store (opt-in, user-controlled): stores the validated accessibility_profile across sessions, together with a history of past route preferences and user corrections. Corrections are explicitly fed back into the profile so that the system improves its defaults for the same user over time without requiring a full re-profiling dialogue. Participation in this tier requires explicit consent, aligned with the UN CRPD's emphasis on user autonomy and data control [UN, 2006].

Environmental Cache (time-bounded, server-side): holds recently fetched WFS responses from Zurich civic data sources. TTL values are calibrated to source update frequencies: barrier and amenity data change slowly (15-minute cache is safe), while live transit feeds require sub-minute refresh. The SafetyAuditorAgent reads directly from this cache to verify plan claims without issuing additional network calls.

This three-tier design separates concerns cleanly: working memory handles transient reasoning state, persistent memory enables personalisation with user consent, and the environmental cache decouples agent reasoning from live network latency.
